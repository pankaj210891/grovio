# Plan 11-06: New Feature Frontends + Cross-cutting

**Phase**: 11 ŌĆö UX/UI Platform Redesign
**Wave**: 5b (requires Wave 5a backend foundations; runs after Waves 2-4 are complete)
**Status**: Ready

## Goal

Implement all new feature frontends that depend on Wave 5a backends (wishlist UI, reviews UI, customer notification center, personalized homepage sections), plus all cross-cutting features that apply across all three portals: dark mode, i18n foundation, WCAG 2.1 AA accessibility pass, performance optimization, animation conventions cleanup, Storybook 8.x component catalog, and social sharing.

## Context

This plan is the final wave of Phase 11. It depends on:
- Wave 5a (11-05) for all new backend endpoints (wishlist, reviews, notifications, personalization)
- Waves 2-4 for portal redesigns (this plan extends them with new feature UIs)
- Wave 1 (packages/ui) for shared components

After this plan completes, Phase 11 is fully done. The cross-cutting tasks (dark mode, i18n, accessibility) touch all three web apps ŌĆö they must be done last to avoid conflicts with portal redesign work in Waves 2-4.

## Dependencies

- Wave 5a (11-05) complete ŌĆö wishlist, reviews, notifications, personalization backend endpoints live
- Waves 2-4 complete ŌĆö all portal pages redesigned so accessibility/dark mode pass covers final state

### T8: Performance optimization

1. **React.lazy + Suspense on all page routes** (if not already done in Waves 2-4):
   - Admin `App.tsx`: lazify all page imports with `React.lazy(() => import(...))`
   - Storefront `App.tsx`: lazify all page imports (vendor `router.tsx` already uses `React.lazy`)
   - Each lazy boundary wraps in `<Suspense fallback={<PageSkeleton />}>` using a shared skeleton from `@grovio/ui`

2. **Image lazy loading**: ensure all `<img>` tags across all three portals have `loading="lazy"` attribute. Ensure all `<img>` tags have explicit `width` and `height` attributes to prevent Cumulative Layout Shift (CLS).

3. **React Query staleTime tuning** ŌĆö apply appropriate stale times by endpoint type:
   - Static/slow-changing data (categories, product details): `staleTime: 10 * 60 * 1000` (10 min)
   - Dashboard KPIs / analytics: `staleTime: 5 * 60 * 1000` (5 min) with `refetchInterval: 60_000` for admin/vendor dashboards
   - Cart / wishlist / notifications: `staleTime: 0` (always fresh, but cached for instant UI)
   - Search results: `staleTime: 2 * 60 * 1000` (2 min)

4. **`packages/ui` tree shaking verification**: confirm that `apps/web-storefront` build output doesn't include unused `@grovio/ui` components. Vite 8 with ESM exports handles this ŌĆö verify with `pnpm --filter @grovio/web-storefront build -- --report`.

### T9: Animation conventions cleanup (D-56)

Apply standardized animation conventions across all three portals:

1. **`useReducedMotion` wrapper**: in all Motion 12.x `motion.div` components across all portals, read `useReducedMotion()` and pass `{ duration: 0 }` or skip `animate` when true. Create a shared `useMotionSafe()` hook in `packages/ui/src/hooks/useMotionSafe.ts` that returns motion props or no-op props based on reduced motion preference.

2. **Easing conventions** (D-56): audit all `motion.div` components and standardize:
   - Entries/reveals: `ease: 'easeOut'`, duration 200-300ms
   - Exits: `ease: 'easeIn'`, duration 150-200ms
   - Drag/pan: `type: 'spring'` with `stiffness: 300, damping: 30`
   - Avoid: duration > 400ms, bouncing springs on non-gesture animations

3. **Remove over-animation** (D-56): remove any Motion animations from: data table row renders, form field focus, standard hover states (use CSS `transition` only), list data that changes frequently.

4. **Page transitions**: ensure `PageTransition.tsx` in storefront is applied consistently on all routes (it may already exist from Phase 4 ŌĆö verify it uses `easeOut` fade+slide 200-300ms).
