# Plan 11-02: Admin Portal Full Redesign

**Phase**: 11 — UX/UI Platform Redesign
**Wave**: 2 (requires Wave 1 packages/ui foundation)
**Status**: Ready

## Goal

Fully redesign the admin panel from a Phase 6 functional dashboard into a command-center experience: command bar, actionable KPI dashboard, analytics insights, unified finance center, KYC-driven vendor management, support ticket system, admin RBAC, bulk actions, and CSV/Excel exports. All new tables and backend endpoints are also created in this wave.

## Context

Phase 6 delivered a working admin panel with: DashboardPage (KPI tiles, revenue chart, top-vendors), VendorsPage (approval/suspend list), CatalogModerationPage, CommissionRulesPage, PayoutManagementPage (separate from CommissionRules), CmsPage, FeatureFlagsPage, SettingsPage, AuditLogPage. These pages are functional but minimal. Phase 11 redesigns them and adds: command bar, Insights, Finance center (merges CommissionRules + PayoutManagement), vendor KYC workflow, support tickets, RBAC enforcement, bulk actions, and exports. Existing Phase 2 category routes are preserved untouched.

## Dependencies

- Wave 1 (`packages/ui`) complete — Shadcn/Radix components available at `@grovio/ui`
- Phase 6 admin backend (vendor approval/suspend, audit_log, commission_entries, vendor_payouts, admin_users table) — no changes to existing logic
- Phase 2 category routes — preserved under same paths, no changes

## Tasks

### T0: Design pass — invoke `/frontend-design` for new admin pages

Before implementing each new page listed below, invoke the `/frontend-design` skill with the page requirements from this plan to generate the initial production-grade styled implementation. The skill produces actual React code with distinctive aesthetics, cohesive typography/color, and purposeful animations — this becomes the implementation skeleton that subsequent tasks wire up with real data.

Pages to design-pass in this wave (run `/frontend-design` for each):

2. **DashboardPage redesign** — clickable KPI tiles, Needs Attention queue, Platform Health widget, Broadcast creation; high-density operational dashboard
3. **InsightsPage** — revenue sparklines, vendor performance table, anomaly flags; analytics-forward, data-dense layout
4. **VendorProfilePage** — KYC workflow, onboarding checklist, performance charts; structured enterprise record view

Design constraint: all pages must feel like a unified admin command center. Use the Grovio design token system (`bg-background`, `text-primary` etc.) from `@grovio/ui` as the foundation. Dark mode aware. Dense information hierarchy with clear visual priority between primary actions and data.

### T1: DB migrations — new admin tables

Run Drizzle migrations to add:

1. `admin_users.role TEXT NOT NULL DEFAULT 'moderator'` — add `role` column to existing `admin_users` table (values: `super_admin`, `moderator`, `finance_admin`)
2. `vendor_kyc_documents` table: `id UUID PK`, `vendor_id UUID FK → vendors`, `document_type TEXT` (id_proof/gst_certificate/bank_verification), `file_url TEXT NOT NULL`, `uploaded_at TIMESTAMPTZ DEFAULT now()`, `verified_at TIMESTAMPTZ nullable`, `verified_by_admin_email TEXT nullable`
3. `announcements` table: `id UUID PK`, `title TEXT NOT NULL`, `body TEXT NOT NULL`, `target_type TEXT NOT NULL` (customers/vendors/all), `active BOOLEAN DEFAULT true`, `expires_at TIMESTAMPTZ nullable`, `created_by_admin_email TEXT NOT NULL`, `created_at TIMESTAMPTZ DEFAULT now()`
4. `support_tickets` table: `id UUID PK`, `subject TEXT NOT NULL`, `body TEXT NOT NULL`, `submitted_by_type TEXT NOT NULL` (customer/vendor), `submitted_by_id TEXT NOT NULL`, `status TEXT NOT NULL DEFAULT 'open'` (pgEnum: open/in_progress/resolved), `assigned_to_admin_email TEXT nullable`, `created_at TIMESTAMPTZ DEFAULT now()`, `updated_at TIMESTAMPTZ DEFAULT now()`
5. `support_ticket_replies` table: `id UUID PK`, `ticket_id UUID FK → support_tickets`, `body TEXT NOT NULL`, `author_type TEXT NOT NULL` (admin/customer/vendor), `author_id TEXT NOT NULL`, `created_at TIMESTAMPTZ DEFAULT now()`

All money-adjacent columns (none in these tables) must use BIGINT minor units per PITFALLS.md Pitfall 1.

### T2: S3-compatible file upload infrastructure

Create `apps/api/src/infrastructure/storage/StorageClient.ts` — a thin wrapper around `@aws-sdk/client-s3` configured for Cloudflare R2 (or Supabase Storage as fallback). Reads endpoint, bucket, access key, secret key from env (`STORAGE_ENDPOINT`, `STORAGE_BUCKET`, `STORAGE_ACCESS_KEY`, `STORAGE_SECRET_KEY`). Expose two functions: `uploadFile(key: string, buffer: Buffer, contentType: string): Promise<string>` (returns public URL) and `getSignedDownloadUrl(key: string): Promise<string>`. Register as Awilix value in `container.ts`. This infrastructure is shared across KYC upload (Wave 2), vendor logo/banner (Wave 3), and product images (Wave 3).

### T3: Backend API — RBAC middleware + admin-only endpoints

1. **RBAC middleware** (`apps/api/src/modules/admin/rbac.middleware.ts`): Fastify preHandler that reads `request.admin.role` and enforces section access. Roles: `super_admin` (all), `moderator` (catalog, vendors, support — no finance, settings), `finance_admin` (finance only — no vendors, settings). Attach to admin route plugins via `preHandler`.

2. **KYC upload endpoint**: `POST /admin/vendors/:id/kyc-documents` — accepts multipart form, validates file type (PDF/JPG/PNG, max 10MB), uploads via StorageClient, inserts into `vendor_kyc_documents`. `PATCH /admin/vendors/:id/kyc-documents/:docId/verify` — sets `verified_at` + `verified_by_admin_email`.

3. **Announcements CRUD**: `POST/GET/PATCH/DELETE /admin/announcements`. `GET /announcements/active` (public endpoint filtered by `target_type` and `expires_at > now() AND active = true`).

4. **Support ticket CRUD**: `GET/PATCH /admin/support-tickets` (list with status/type filter), `GET /admin/support-tickets/:id` (full detail with replies), `POST /admin/support-tickets/:id/replies`, `PATCH /admin/support-tickets/:id` (status, assignment). Log status transitions to `audit_log` (`support_ticket.status_changed` action).

5. **Insights endpoints** (`GET /admin/insights/revenue`, `/vendor-performance`, `/product-velocity`, `/retention-rate`, `/anomaly-flags`) — each queries PostgreSQL aggregations with Redis 1-hour TTL cache following `FeatureFlagService` cache pattern.

6. **Platform health endpoint**: `GET /admin/health/metrics` — returns BullMQ queue depths (from Bull Board API or direct ioredis query), OpenSearch last sync timestamp from a `system_metadata` store, and a recent API health check latency.

7. **Export endpoints**: `GET /admin/orders/export?format=csv`, `GET /admin/vendors/export?format=csv`, `GET /admin/finance/commissions/export?format=csv`, `GET /admin/finance/payouts/export?format=csv`, `GET /admin/audit-log/export?format=csv` — each streams CSV using Node.js `stream.PassThrough`. No RBAC exception: finance exports require `finance_admin|super_admin`.

8. **Bulk product import**: `POST /admin/products/import` — multipart CSV/XLSX upload, parse rows, validate required columns (name, category_id, price_minor, quantity), batch-insert into `products` table with `status = 'pending'`. Return per-row validation errors. `GET /admin/products/import/template` returns a downloadable template CSV.

## Notes

- Phase 2 category routes (`/categories`, `/categories/:id`) must not be touched — they are preserved exactly as-is
- CommissionRulesPage and PayoutManagementPage become dead routes after FinancePage takes over — remove their route registrations and delete the page files
- `finance_admin` role is read-only on Commissions/Refunds tabs (no settle or adjust actions) — only `super_admin` can trigger settlements
- StorageClient env vars (`STORAGE_ENDPOINT` etc.) must be added to `apps/api/.env.example`
- Bull Board is dev-only; platform health widget falls back to "metrics unavailable" if the Bull Board API route is not mounted in production
