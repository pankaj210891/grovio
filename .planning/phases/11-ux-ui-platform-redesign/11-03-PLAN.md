# Plan 11-03: Vendor Portal Full Redesign

**Phase**: 11 — UX/UI Platform Redesign
**Wave**: 3 (requires Wave 1 packages/ui; Wave 2 S3 StorageClient must be complete for logo/banner upload)
**Status**: Ready

## Goal

Redesign the vendor portal from a basic Phase 6 functional shell into a workflow-first, morning-glance experience. Deliver: morning-glance dashboard with stat tiles and inventory alerts, multi-step product creation wizard with draft saving, order kanban with bulk ship, vendor finance center with drill-down tabs, analytics section with product performance and inventory forecasting, enhanced store profile with image upload and return policy editor, improved team management, and a persistent onboarding checklist.

## Context

The vendor portal (`apps/web-vendor`) has routing defined in `router.tsx` and all page files exist from Phase 6: DashboardPage, ProductsPage, InventoryPage, OrdersPage, ReturnsPage, EarningsPage, CouponsPage, TeamPage, StoreProfilePage, SettingsPage. These pages are functional but minimal. Phase 11 fully redesigns each page and adds new backend endpoints for analytics and finance. `App.tsx` is a placeholder — `main.tsx` correctly uses `router.tsx`. The vendor portal is well-structured: PanelLayout, ProtectedVendorRoute (with `requiredRole` support), Sidebar, Header already exist.

## Dependencies

- Wave 1 (`packages/ui`) complete — Shadcn components available at `@grovio/ui`
- Wave 2 StorageClient complete — file upload for logo/banner/KYC reuses same infrastructure
- Phase 6 backend: vendor commission entries, vendor payouts, inventory/pricing endpoints, orders endpoints
- Phase 5: `vendor_return_policies` table and API (Phase 11 adds vendor UI to configure it)

## Tasks

Before implementing each redesigned or new page below, invoke the `/frontend-design` skill with the page requirements to generate the initial production-grade styled implementation. The skill produces actual React code with cohesive aesthetics, purposeful motion, and production polish — this becomes the implementation skeleton that subsequent tasks wire up with real data.

Pages to design-pass in this wave:
1. **DashboardPage redesign** — morning-glance stat tiles (6), dual-column sparklines/alerts, 60s polling indicators; fast-decision, at-a-glance vendor operations dashboard
2. **CreateProductWizard** — 3-step flow with step indicators, dynamic attribute fields, drag-to-reorder images, variant matrix; guided, clear wizard aesthetic
3. **OrdersPage redesign** — list/kanban toggle, 5-column kanban board, bulk ship modal; order operations center

Consistent with the admin portal's dark mode capability. Compact information density appropriate for a power user who manages their store daily.

### T1: Backend API — vendor analytics + finance endpoints

Add new endpoints to the vendor API module in `apps/api/src/modules/vendor/`:

1. **Analytics endpoints** (all scoped by `request.vendor.id`, Redis-cached 15 min):
   - `GET /vendor/analytics/product-performance` — per-product: view count (from `customer_product_views` Wave 5a), add-to-cart count, orders, revenue_minor, return rate, wishlist count. Returns sortable array.
   - `GET /vendor/analytics/inventory-forecast` — per-product: current stock, avg daily sales (30d), estimated days of stock remaining
   - `GET /vendor/analytics/conversion-funnel` — impressions → views → add-to-cart → orders with drop-off %
   - `GET /vendor/analytics/customer-behavior` — new vs returning customer split, avg order value trend, peak purchase hours/days (from `orders` table)
   - Note: `customer_product_views` and `wishlists` tables are created in Wave 5a. These endpoints should gracefully return empty data if those tables don't yet exist (check table existence or use optional joins).

2. **Finance endpoints** (scoped by `request.vendor.id`):
   - `GET /vendor/finance/overview?period=7d|30d|90d` — aggregates: gross_sales_minor, platform_fees_minor, net_revenue_minor, total_settled_minor, pending_payout_minor, tax_withheld_minor (all BigInt in minor units, serialized as strings in JSON)
   - `GET /vendor/finance/transactions?period=...` — order-level breakdown: per-order gross → fee → net
   - `GET /vendor/finance/settlements` — existing `vendor_payouts` records
   - `GET /vendor/finance/tax-summary?period=...` — monthly tax breakdown
   - `GET /vendor/finance/transactions/export?format=csv`, `GET /vendor/finance/settlements/export?format=csv`

3. **Return policy endpoint** (Phase 5 deferred completion):
   - `GET /vendor/return-policy` — returns vendor's current `vendor_return_policies` row
   - `PUT /vendor/return-policy` — upserts the return policy record

4. **Announcement endpoint**: `GET /announcements/active?target=vendor` — reuses Wave 2 backend (filter `target_type IN ('vendors','all')`).

5. **Onboarding status endpoint**: `GET /vendor/onboarding-status` — returns checklist state: profile_complete, bank_details_added, kyc_submitted, first_product_approved, approval_requested. Computed from vendor record state.

### T3: Multi-step product creation wizard (ProductsPage.tsx + CreateProductWizard)

The existing ProductsPage shows a product list. Add a "Create Product" button that opens/navigates to a multi-step wizard.

1. **Product list** (keep existing table structure but migrate to `DataTable` from `@grovio/ui`): columns — name, category, status badge, stock, price, last updated. Bulk action bar: Archive Selected. Filter by status.

2. **CreateProductWizard** (`apps/web-vendor/src/components/products/CreateProductWizard.tsx`):
   - Rendered at `/products/new` as a full-page wizard
   - Uses React state or Zustand local store for wizard step state
   
   **Step 1 — Category & Attributes**:
   - Category select (loads all categories from `GET /categories`)
   - On category select, loads attribute template from `GET /categories/:id/attributes`
   - Renders attribute fields dynamically based on attribute type (text input, select, color swatch, etc.)
   - Product name, description (rich text: textarea with markdown preview)
   
   **Step 2 — Images & Variants**:
   - Image upload: drag-drop multi-image uploader → upload via StorageClient → stores URLs
   - Drag-to-reorder images (Framer Motion drag or HTML5 drag API)
   - Variant builder: if product has color/size attributes, show variant matrix; each variant gets its own stock quantity
   - SKU auto-generation: `{PRODUCT_SLUG}-{VARIANT_COMBO}` — shown in an editable field
   
   **Step 3 — Pricing & Publish**:
   - Price input (converted to minor units on submit), compare-at price, cost price
   - Inventory quantity per variant (or single if no variants)
   - Low stock threshold input
   - Publish toggle: Draft / Published
   - "Save as Draft" and "Publish" buttons
   
3. **Draft saving**: After Step 1 completes, `POST /vendor/products` with `status = 'draft'` creates the product. Steps 2 and 3 call `PATCH /vendor/products/:id`. Product ID stored in wizard state for continuation.

4. **Bulk CSV upload**: Secondary upload option on ProductsPage — links to the admin CSV import infrastructure (same endpoint accessible to vendors for their products via `POST /vendor/products/import`).

5. **InventoryPage.tsx**: Merge inventory management (stock, price updates) into a dedicated tab on ProductsPage detail. Update `router.tsx` to keep `/inventory` route but have it redirect to `/products?tab=inventory` or render an inventory-focused view.

## Notes

- `App.tsx` in web-vendor is a placeholder — it is NOT used. `main.tsx` imports from `router.tsx`. Do not touch `App.tsx`.
- `InventoryPage.tsx` becomes redundant after product wizard integrates inventory; keep the file but redirect the route to `/products`
- Recharts is the preferred chart library (lightweight, React 19 compatible). Add `recharts` to `apps/web-vendor` dependencies if not already present.
- All money values from finance endpoints are BIGINT minor units serialized as strings — frontend must parse with `BigInt()` or divide by 100 for display, never use `parseFloat`
- Wave 5a (new feature backends) must be complete before analytics endpoint returns real data — vendor analytics page must gracefully handle empty/zero state until then
