# Plan 11-04: Customer Storefront Redesign

**Phase**: 11 — UX/UI Platform Redesign
**Wave**: 4 (requires Wave 1 packages/ui foundation; no Wave 2/3 dependency)
**Status**: Ready

## Goal

Redesign the customer storefront from Phase 4's functional foundation into a conversion-optimized, mobile-first experience. Deliver: mobile bottom navigation, sticky collapsible PLP filters with view toggle and comparison tray, rich PDP gallery with variant swatches and delivery estimation, enhanced search with history and popular queries, cart coupon suggestions and recommendations, one-page checkout accordion (replaces multi-step flow), visual order timeline with reorder, and SEO infrastructure with per-page meta tags and JSON-LD.

## Context

The customer storefront (`apps/web-storefront`) has a solid Phase 4 foundation: all core routes exist (HomePage, CategoryPage/PLP, ProductDetailPage/PDP, CartPage, SearchPage, multi-step checkout, account pages), working auth/React Query/Zustand setup, FilterSidebar, ProductCard, SearchBar, PlacesAutocompleteInput. Phase 11 is a targeted iteration — keep the routing structure and data-fetching infrastructure, redesign page layouts and interaction patterns, and add missing features. The multi-step checkout URL structure (`/checkout/address → /checkout/delivery → /checkout/payment → /checkout/review`) is replaced by a single `/checkout` route with accordion.

New features that require new DB tables (wishlist, reviews, notifications) are NOT in this plan — those backends and frontends are in Waves 5a/5b. This plan focuses on the visual redesign and features that work with existing data.

## Dependencies

- Wave 1 (`packages/ui`) complete — Shadcn components, `cn()` utility available
- Phase 4 storefront foundation (routing, auth, React Query client, Zustand stores) — extended, not replaced
- Phase 5 commerce backend (basket, checkout, coupons, orders, return_requests) — extended with new endpoints below

### T7: One-page checkout accordion (replaces multi-step flow)

Replace the Phase 5 multi-step checkout (`/checkout/address`, `/checkout/delivery`, `/checkout/payment`, `/checkout/review`) with a single `/checkout` route using an accordion.

1. **New CheckoutPage.tsx** (`apps/web-storefront/src/pages/checkout/CheckoutPage.tsx`):
   - Single route at `/checkout`
   - Uses `Accordion` from `@grovio/ui` (Radix-based)
   - 4 accordion sections that expand in sequence:
     - **Address**: PlacesAutocompleteInput + saved address selector (if authenticated). "Continue" button validates + advances to Delivery
     - **Delivery**: slot/method selector. "Continue" advances to Payment
     - **Payment**: Stripe/Razorpay payment form (reuse existing `StripePaymentForm.tsx` and `RazorpayButton.tsx`). "Continue" advances to Review
     - **Review**: order summary + place order button
   - Each section shows a compact summary when collapsed (address text, delivery method, payment method last 4)
   - Back within checkout: clicking a collapsed section re-opens it and collapses subsequent sections
   - Guest checkout: "Continue as Guest" path clearly visible at top of page; email capture in Review section for guests

2. **CheckoutGuard.tsx** (`apps/web-storefront/src/components/checkout/CheckoutGuard.tsx`): already exists — update to accept `redirectTo="/checkout"` instead of the old multi-step URLs.

3. **Update App.tsx routing**: Replace the 4 checkout route registrations (`/checkout/address`, `/checkout/delivery`, `/checkout/payment`, `/checkout/review`) with a single `/checkout` route pointing to `CheckoutPage`. Add redirects from old paths to `/checkout` for any bookmarked links.

4. The existing `CheckoutProgress.tsx` component is no longer needed — delete it. The accordion provides visual progress.

### T9: Backend endpoints for Wave 4 features

Add these endpoints to `apps/api/`:

1. `GET /serviceability?pincode=XXX` (ProductRoutes.ts): returns `{ available: boolean, estimatedDays: number, message: string }`. For v1, return a stub: all pincodes are serviceable, estimated 3-5 days. The stub is acceptable for v1 demo.

2. `GET /search/popular` (SearchRoutes.ts): returns `string[]` of top 10 search terms. Read from Redis `popular_searches` key (populated by a simple daily BullMQ job that aggregates recent search queries). If key missing, return an empty array.

3. `GET /coupons/available` (CouponRoutes.ts): reads `cart_total_minor` and `product_category_ids[]` from query params. Returns applicable coupons filtered by: `expires_at > now()`, `usage_count < usage_limit` (or unlimited), `min_order_amount_minor ≤ cart_total_minor`, `scope = 'all' OR scope category intersects product_category_ids`.

4. `GET /products/recommendations/cart` (ProductRoutes.ts): accepts `category_ids[]` query param. Returns up to 6 products from those categories (excluding IDs in `exclude_ids[]` param). Redis-cached 30 min per category set key.

5. Update `GET /search/suggestions` to include `categoryName: string` on each suggestion item.

### T10: SEO — react-helmet-async + JSON-LD

1. Install `react-helmet-async` in `apps/web-storefront`.

2. Wrap `apps/web-storefront/src/main.tsx` root with `<HelmetProvider>`.

3. **Per-page meta tags** using `<Helmet>` in each page component:
   - HomePage: `<title>Grovio — Online Marketplace</title>` + description
   - CategoryPage: `<title>{category.name} | Grovio</title>` + category description
   - ProductDetailPage: `<title>{product.name} | Grovio</title>` + product short description + OG image from first product image
   - SearchPage: `<title>Search: {query} | Grovio</title>`
   - Account pages: `<title>My Account | Grovio</title>` (noindex)

4. **JSON-LD Product schema** on PDP:
   ```json
   {
     "@context": "https://schema.org",
     "@type": "Product",
     "name": "...",
     "image": ["..."],
     "offers": { "@type": "Offer", "price": "...", "priceCurrency": "INR", "availability": "..." },
     "aggregateRating": { "@type": "AggregateRating", "ratingValue": 0, "reviewCount": 0 }
   }
   ```
   Inject via `<script type="application/ld+json">` inside `<Helmet>`.

5. **BreadcrumbList JSON-LD** on CategoryPage.

6. **Sitemap generation script** (`apps/web-storefront/scripts/generate-sitemap.ts`): Node.js script that fetches all category slugs and first 1000 product slugs from the API, writes `public/sitemap.xml`. Add `generate:sitemap` to `package.json` scripts.
