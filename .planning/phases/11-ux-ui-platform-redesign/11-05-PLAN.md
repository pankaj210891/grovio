# Plan 11-05: New Feature Backends

**Phase**: 11 — UX/UI Platform Redesign
**Wave**: 5a (can run parallel with Waves 2-4; provides backend foundations for Wave 5b frontends)
**Status**: Ready

## Goal

Build all new backend infrastructure required by Phase 11's new features: wishlist, product reviews, customer notifications, personalization (view history + trending + recommendations + personalized search ranking), support tickets, announcements, and PDF invoice generation. All new tables follow project conventions (BIGINT minor units, pgEnum for status, Awilix DI, Fastify plugins). Wave 5b frontends depend on this plan being complete.

## Context

Waves 2, 3, and 4 redesign existing pages using existing backend data. Wave 5a creates all the new DB tables and API endpoints that the new feature frontends (wishlist, reviews, notification center, personalized homepage, price drop alerts) require. These tables are referenced in CONTEXT.md: `wishlists`, `product_reviews`, `customer_notifications`, `customer_product_views`, `support_tickets`, `support_ticket_replies`, `announcements` (already planned in Wave 2 admin UI — backend endpoint is shared). PDF invoice generation is backend-only in this wave; frontend download button is in Wave 4.

## Dependencies

- Phase 5 commerce backend: `orders`, `order_items`, `basket_items`, `coupons`, `products` tables — all read by new endpoints
- Phase 6 backend: `vendor_commission_entries`, `vendor_payouts`, `audit_log` — referenced but not modified
- Wave 2 plan (11-02) DB migrations must have run the `announcements` table before T1 if running sequentially; if parallel, include `announcements` in this plan's migrations or coordinate
- BullMQ job infrastructure from Phase 3/6 — new jobs follow same pattern in `apps/api/src/modules/jobs/`
- OpenSearch infrastructure from Phase 3 — personalized search ranking adds a `function_score` wrapper to existing queries

## Tasks

### T1: DB migrations — wishlists, reviews, notifications, views

Create Drizzle migration files adding:

1. **`wishlists`** table: `id UUID PK`, `customer_id UUID FK → customers NOT NULL`, `product_id UUID FK → products NOT NULL`, `created_at TIMESTAMPTZ DEFAULT now()`. Unique constraint: `(customer_id, product_id)`. No soft-delete (hard delete is fine per CONTEXT.md).

2. **`product_reviews`** table: `id UUID PK`, `customer_id UUID FK → customers NOT NULL`, `product_id UUID FK → products NOT NULL`, `rating SMALLINT NOT NULL CHECK(rating BETWEEN 1 AND 5)`, `title TEXT`, `body TEXT NOT NULL`, `verified_purchase BOOLEAN NOT NULL DEFAULT false`, `vendor_reply TEXT nullable`, `vendor_replied_at TIMESTAMPTZ nullable`, `moderated BOOLEAN NOT NULL DEFAULT false`, `moderated_by_admin_email TEXT nullable`, `created_at TIMESTAMPTZ DEFAULT now()`, `updated_at TIMESTAMPTZ DEFAULT now()`. Unique constraint: `(customer_id, product_id)`.

3. **`customer_notifications`** table: `id UUID PK`, `customer_id UUID FK → customers NOT NULL`, `type TEXT NOT NULL` (pgEnum: order_update/price_drop/return_update/promotion), `title TEXT NOT NULL`, `body TEXT NOT NULL`, `reference_id TEXT nullable` (e.g., order ID), `reference_type TEXT nullable` (order/product/return), `dismissed_at TIMESTAMPTZ nullable`, `created_at TIMESTAMPTZ DEFAULT now()`. Append-only (no UPDATE to existing rows — use `dismissed_at` column for soft-hide).

4. **`customer_product_views`** table: `id UUID PK`, `customer_id UUID FK → customers NOT NULL`, `product_id UUID FK → products NOT NULL`, `viewed_at TIMESTAMPTZ DEFAULT now()`. Index: `(customer_id, viewed_at DESC)` for Recently Viewed query. Index: `(product_id, viewed_at)` for trending computation. No unique constraint (multiple views by same customer are recorded).

5. Add **`avg_rating NUMERIC(3,2) DEFAULT 0`** and **`review_count INTEGER DEFAULT 0`** columns to `products` table — cached aggregates updated on review insert/update/delete.

6. Add **`sold_count INTEGER DEFAULT 0`** column to `products` table — updated on order placed (BullMQ job or order service hook).

All columns referencing money must use `_minor BIGINT` naming. None in these tables require it.

### T2: Wishlist API

Create `apps/api/src/modules/wishlist/` Fastify plugin:

- `POST /wishlist/:productId` — authenticates customer (httpOnly cookie auth per Phase 4 D-09), upserts into `wishlists` (ignore duplicate due to unique constraint). Returns `{ wishlisted: true }`.
- `DELETE /wishlist/:productId` — deletes row for `(customer_id, product_id)`. Returns `{ wishlisted: false }`.
- `GET /account/wishlist` — returns paginated list of wishlisted products (join to `products` table) with `is_price_dropped: boolean` flag (compare current price to price at wishlist creation time — store `price_at_wishlist_minor BIGINT` column in `wishlists` migration).
- `GET /wishlist/status?productIds=id1,id2,...` — returns a map `{ [productId]: boolean }` of wishlist status for the given IDs (used to initialize ProductCard hearts on page load).

Add `price_at_wishlist_minor BIGINT NOT NULL DEFAULT 0` column to `wishlists` migration above. Populate on INSERT by reading `products.price_minor` at insert time.

Register plugin in `apps/api/src/app.ts`.

### T3: Product reviews API

Create `apps/api/src/modules/reviews/` Fastify plugin:

- `POST /products/:id/reviews` — authenticated customer only. Validates `verified_purchase` by checking if customer has an order containing this product (query `order_items`). Inserts into `product_reviews`. On insert: update `products.avg_rating` and `products.review_count` via aggregation query.
- `GET /products/:id/reviews?page=1&limit=10` — public endpoint. Returns paginated reviews ordered by `created_at DESC`. Includes vendor reply if present. Returns 404 if product not found.
- `PATCH /vendor/reviews/:reviewId/reply` — vendor-only. Sets `vendor_reply` + `vendor_replied_at`. Validates that review belongs to a product the vendor owns.
- `DELETE /admin/reviews/:reviewId` — admin-only. Sets `moderated = true` + `moderated_by_admin_email`. Soft-hides from public API (filter `WHERE moderated = false`).

Register plugin in `apps/api/src/app.ts`.

### T4: Customer notifications API + price drop BullMQ job

Create `apps/api/src/modules/notifications/customer/` Fastify plugin:

- `GET /account/notifications?page=1&limit=20` — authenticated customer. Returns notifications ordered by `created_at DESC`. `dismissed_at IS NULL` first, then dismissed.
- `PATCH /notifications/:id/dismiss` — sets `dismissed_at = now()` for the notification.
- `GET /account/notifications/preferences` — returns customer's notification preferences from a new `customer_notification_preferences` table or a JSONB column on `customers`. Fields: `order_updates BOOLEAN DEFAULT true` (always on, not editable), `price_drops BOOLEAN DEFAULT true`, `promotions BOOLEAN DEFAULT false`.
- `PATCH /account/notifications/preferences` — updates `price_drops` and `promotions` preference (order_updates immutable).

**PriceDropCheckJob** (`apps/api/src/modules/jobs/PriceDropCheckJob.ts`):
- Triggered when vendor updates product price (`PATCH /vendor/products/:id/pricing`)
- If new price < old price: query all customers who have product in wishlist (`wishlists` where `product_id = id` and `price_at_wishlist_minor > new_price_minor`)
- For each such customer: insert a `customer_notifications` row of type `price_drop` with `reference_id = productId`, `reference_type = 'product'`
- Also update `wishlists.price_at_wishlist_minor = new_price_minor` for all rows
- BullMQ job registered in `apps/api/src/modules/jobs/index.ts`

### T5: Personalization backend — view history + trending + recommendations

Create `apps/api/src/modules/personalization/` Fastify plugin:

1. **`POST /products/:id/view`** — authenticated customer: insert into `customer_product_views`. Guest: no-op (guest localStorage tracking is frontend-only). Rate-limit: if same `(customer_id, product_id)` viewed within last 5 minutes, skip insert (avoid duplicate counts on page refresh).

2. **`GET /products/recently-viewed`** — authenticated customer: returns last 10 distinct products from `customer_product_views` ordered by `viewed_at DESC`.

3. **`GET /products/trending`** (D-47): weighted score = `(order_count_7d × 3) + (view_count_7d × 1)` per product. Queries `order_items` + `customer_product_views` for last 7 days, orders by score DESC, LIMIT 20. Redis-cached with 1-hour TTL key `trending:products`. Public endpoint (no auth required).

4. **`GET /products/:id/frequently-bought-together`** (D-48): co-purchase query — finds products frequently ordered together with this product from `order_items`. Groups by co-occurring `product_id` in same `order_id`, orders by frequency DESC, LIMIT 6. Redis-cached per product ID with 2-hour TTL.

5. **`GET /products/related/:id`** (D-48): same category + similar price range (±30% of current price). Returns up to 8 products ordered by `avg_rating DESC`. Redis-cached per product ID with 1-hour TTL.

6. **`GET /homepage/personalized`** (D-03): authenticated customer endpoint. Returns:
   - `recentlyViewed`: last 10 products from `customer_product_views`
   - `continueShoppingCategory`: last browsed category from `customer_product_views` (most recent product's category)
   - `recommendations`: top 6 products from categories the customer has ordered from (not yet purchased again). If insufficient order history, fallback to trending products.
   - All Redis-cached per `customer_id` with 15-min TTL.

7. **Personalized search ranking** (D-49): Update the OpenSearch search query in `SearchService.ts` to wrap the existing query in a `function_score` that boosts results from categories the customer has purchased from. Boost value: 1.5×. Only applied when request has authenticated customer JWT. Graceful no-op for guest searches.

### T6: Support tickets API (admin + customer/vendor submission)

Support ticket backend was planned in Wave 2 (T3.4) as admin-only management. Extend here with customer/vendor submission:

- `POST /support/tickets` — authenticated customer or vendor. Creates ticket with `submitted_by_type` and `submitted_by_id` from auth context.
- `GET /account/support-tickets` — returns all tickets submitted by this customer.
- `POST /account/support-tickets/:id/replies` — customer replies to their own ticket.

These endpoints complement the admin support ticket endpoints from Wave 2 T3.4. Register in a shared `support` plugin or extend Wave 2's plugin.

### T7: PDF invoice endpoint

Install `pdfmake` (zero-native-binary PDF library) in `apps/api`:

```
pnpm --filter apps/api add pdfmake
pnpm --filter apps/api add -D @types/pdfmake
```

Create `apps/api/src/modules/orders/InvoiceService.ts`:
- `generateInvoicePdf(orderId: string, customerId: string): Promise<Buffer>` — fetches order + items + customer from DB, builds a `pdfmake` document definition with:
  - Grovio header (store name, logo if configured via `marketplace_settings`, date)
  - Order ID, placed date, billing address
  - Items table: name, qty, unit price (minor units ÷ 100 for display), line total
  - Taxes line, order total, platform fee line (if applicable)
  - Footer: "Thank you for your order" + vendor store name + contact
- Returns a PDF `Buffer`

New endpoint in OrderRoutes.ts:
- `GET /orders/:id/invoice` — authenticated customer. Validates order belongs to customer. Calls `InvoiceService.generateInvoicePdf()`. Sets `Content-Type: application/pdf` and `Content-Disposition: attachment; filename="invoice-{orderId}.pdf"`. Streams Buffer as response.

Admin version: `GET /admin/orders/:id/invoice` — admin can download any order's invoice.

### T8: Popular searches endpoint + sold_count updates

1. **`GET /search/popular`** (Wave 4 T9.2): Implemented here as backend endpoint. A BullMQ `PopularSearchesJob` runs daily: aggregates top 10 search query strings from a `search_query_log` table (or Redis sorted set `search:queries` incremented on each search), stores result in Redis key `search:popular` with 24h TTL. Create `search_query_log` table or use Redis ZINCRBY pattern. `GET /search/popular` reads from Redis key, returns `string[]`.

2. **`sold_count` updates**: On order placed (in `OrderService.createOrder()`), increment `products.sold_count` for each product in the order. Use a single `UPDATE products SET sold_count = sold_count + quantity WHERE id = product_id` per line item. No BullMQ needed — synchronous update in the order creation transaction.

### T9: Admin product view analytics (Insights panel extension)

Extend `GET /admin/insights/` endpoints from Wave 2 T3.5 to include:

- **`GET /admin/insights/product-views`**: most-viewed products platform-wide (last 7d) — from `customer_product_views` grouped by `product_id`, ordered by count DESC, LIMIT 20. Include product name + category.
- **`GET /admin/insights/view-conversion-gap`**: products with high view count but low order conversion (last 7d). Formula: `view_count > 50 AND (order_count / view_count) < 0.02`. These are candidates for pricing or UX review. Returns up to 20 items.

Both Redis-cached with 1-hour TTL. These feed the admin Insights panel extension in Wave 5b.

## Success Criteria

1. All Drizzle migrations run without error on a fresh DB; `pnpm --filter @grovio/api db:migrate` passes
2. `POST /wishlist/:productId` correctly inserts/upserts; `GET /account/wishlist` returns correct products with `is_price_dropped` flag set correctly when price has dropped
3. `POST /products/:id/reviews` correctly sets `verified_purchase = true` only when customer has an order containing the product; `products.avg_rating` updates after review insert
4. `POST /products/:id/view` inserts a view record; rate-limiting prevents duplicate inserts within 5 minutes
5. `GET /products/trending` returns up to 20 products ordered by weighted score; result is Redis-cached (second call returns from cache, no DB query)
6. `GET /products/:id/frequently-bought-together` returns products that co-appear in orders with the given product
7. `GET /orders/:id/invoice` returns a valid PDF with correct Content-Type header and correct line items
8. PriceDropCheckJob: when a product price is lowered below `price_at_wishlist_minor`, a `customer_notifications` row is inserted for each customer with that product wishlisted
9. `GET /account/notifications` returns notifications in correct order; `PATCH /notifications/:id/dismiss` sets `dismissed_at`
10. `GET /homepage/personalized` returns all 3 sections (recentlyViewed, continueShoppingCategory, recommendations) for an authenticated customer with order history

## Notes

- All money display in PDF invoice uses minor units ÷ 100 formatted with 2 decimal places — never parseFloat on BIGINT
- `pdfmake` is chosen over PDFKit because it requires no native binaries (PDFKit is also acceptable if pdfmake has issues — both are listed as options in CONTEXT.md D-59)
- The `customer_product_views` table has no unique constraint — multiple views per (customer, product) are valid for accurate trending computation
- Personalized search ranking (T5.7) modifies `SearchService.ts` — the `function_score` must degrade gracefully: if no customer order history, skip boosting and return standard ranked results
- `wishlists.price_at_wishlist_minor` enables price drop detection without needing to store historical price data in a separate table
