<objective>
Build the checkout wizard (4 URL-addressable steps with a skip-guard D-05), the payment step (Stripe Elements + Razorpay modal + hybrid wallet D-09/D-13/WAL-05), order confirmation (ORD-01), and the account order history/detail + wallet pages (ORD-03/04, WAL-01/02). <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Stripe/Razorpay | Card data entered in provider-hosted iframes/modals (PCI scope stays with the provider). |
| browser → checkout API | Customer submits address/provider/wallet selections; amounts are advisory only. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04 | Tampering | displayed totals + wallet amount | mitigate | All totals come from GET /checkout/summary; the wallet amount the UI sends is re-capped server-side (computeWalletApplied). The client cannot set the authoritative charge (CHK-04, WAL-05). |
| T-05-PCI | Information Disclosure | card entry | mitigate | Stripe PaymentElement (iframe) and Razorpay modal keep card data in the provider's PCI scope; the app never handles raw card numbers. |
| T-05-06 | Elevation of Privilege | checkout/account pages | mitigate | Checkout + account routes are under ProtectedRoute (customer auth); order/wallet reads are customer-scoped server-side. |
| T-05-DSK | Tampering | step skipping | mitigate | CheckoutGuard derives step completion from React Query/checkout state and redirects on unmet prerequisites (D-05). |
</threat_model>
