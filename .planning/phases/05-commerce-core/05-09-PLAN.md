<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → checkout | Client submits a checkout request; amounts in the payload are untrusted. |
| webhook → finalizeOrder | Payment confirmation triggers order finalization (may arrive twice or early). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04 | Tampering | CheckoutService.computeSummary | mitigate | All prices re-fetched from catalog; coupon discount recomputed; wallet applied derived server-side. No client amount is trusted (CHK-04, Pitfall 5/6). |
| T-05-01 | Tampering | initiateCheckout reservation | mitigate | Reservation via InventoryService row-lock at proceed-to-payment (CHK-05, D-06), not on basket add. |
| T-05-02 | Tampering | finalizeOrder | mitigate | finalizeOrder idempotent (no-op when already payment_received) and keyed off providerOrderId — handles duplicate/early webhook delivery (Pitfall 8). |
| T-05-PAY | Information Disclosure | SDK coupling | mitigate | CheckoutService uses PaymentService abstraction; no SDK import (PAY-01, Pitfall 9). |
</threat_model>
