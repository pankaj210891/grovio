Apply the database migration that makes the Phase 2 schema live, then build CategoryService — the heart of the category engine. CategoryService mirrors FeatureFlagService's Redis-first pattern and adds: a domain-level 3-level depth guard (D-01), an in-memory adjacency-list-to-nested-tree builder, slug generation with collision handling, soft-delete archive, sibling reorder, and write-through cache invalidation after every mutation (D-03).

Purpose: The migration is the canonical proof the DB schema is live (typecheck passes before migration runs, creating a false-positive — this gate closes that gap). CategoryService owns the depth rule so the error message is meaningful and impossible to bypass via a new route.
Output: Applied migration + CategoryService + unit tests covering CAT-01 and CAT-02.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06 | Tampering | depth bypass via direct insert | mitigate | Depth guard runs in CategoryService.createCategory before the DB insert; routes (plan 02-06) always go through the service, never raw db.insert (Pitfall 3) |
| T-02-07 | Tampering | slug injection / collision | mitigate | slugify strips non-URL-safe chars; service checks uniqueness and appends -2/-3 on collision returning a clean error instead of a raw 500 DB constraint error (Pitfall 7) |
| T-02-08 | Information Disclosure | stale tree after admin write | mitigate | Write-through invalidation inside every mutation method (Pitfall 2); unit test asserts redis.del("cat:tree") after each mutation |
| T-02-SC | Tampering | npm installs | accept | No package installs — all deps already in apps/api; legitimacy gate handled in plan 02-07 |
