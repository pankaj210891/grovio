## ID Strategy

### Question 1: Primary key type

| Option | Description | Selected |
|--------|-------------|----------|
| Keep UUID strings (Recommended) | Store UUIDs as string _id fields. packages/contracts ID types stay unchanged. No breaking change to any existing API response. | ✓ |
| Switch to MongoDB ObjectId | Use ObjectId as _id. Must update packages/contracts ID types, all route params, all web-admin API calls, Phase 3+ cross-references. | |
| UUID stored as Binary (UUID subtype) | MongoDB Binary UUID format for optimal index performance. Rarely worth the complexity at this scale. | |

**User's choice:** Keep UUID strings
**Notes:** Stability of packages/contracts and web-admin compatibility were the deciding factors.

---

### Question 2: UUID generation responsibility

| Option | Description | Selected |
|--------|-------------|----------|
| Service layer generates UUID (Recommended) | Each service calls crypto.randomUUID() before insert. Explicit, testable, no library dependency. | ✓ |
| Mongoose pre-save hook generates UUID | Add a pre('save') hook in the Mongoose schema. Central but implicit — harder to trace in tests. | |
| Utility function in packages/contracts | Export generateId() from contracts. Adds a cross-package dependency for a one-liner. | |

**User's choice:** Service layer generates UUID
**Notes:** Consistent with the existing Drizzle pattern and keeps generation visible in the call site.

---

### Question 3: Cross-collection reference field typing

| Option | Description | Selected |
|--------|-------------|----------|
| Plain String type, no ObjectId ref (Recommended) | Define category_id as { type: String }. No .populate(), services join by querying directly. Simplest. | ✓ |
| Schema.Types.ObjectId with ref but stored as String | Technically incorrect — Mongoose would try to cast strings to ObjectId on queries. | |
| You decide | Leave ref typing to the planner. | |

**User's choice:** Plain String type, no ObjectId ref

### Question 2: How services access models

| Option | Description | Selected |
|--------|-------------|----------|
| Import models directly in services (Recommended) | Services import CategoryModel etc. directly from db/models/. Standard Mongoose pattern. Removes `db` from DI. | ✓ |
| Inject models via Awilix DI | Register each model in the container. Unusual for Mongoose — adds complexity for no benefit. | |

**User's choice:** Import models directly in services
**Notes:** Mongoose models are module-level singletons — DI injection adds nothing here.

## Field Naming Conventions

| Option | Description | Selected |
|--------|-------------|----------|

## Claude's Discretion

- **mongoose.test.ts** — whether to write a plugin connectivity test or skip and rely on service test coverage
- **JSONB field schema strictness** — Schema.Types.Mixed vs inline sub-schemas for options[] and blocks[] fields
- **Orphan-subtree integrity enforcement** — service layer vs Mongoose pre-remove hook for the PostgreSQL ON DELETE RESTRICT equivalent

## Deferred Ideas

- **MongoDB replica set for transactions** — Phase 5 (Commerce Core) needs multi-document ACID transactions (inventory reservation + order placement). Phase 2.1 only has single-document operations. Evaluate replica set requirement before Phase 5 planning.
- **Connection pool tuning** — Default Mongoose pool (5 connections) sufficient for Phase 2.1. Tune maxPoolSize/minPoolSize/serverSelectionTimeoutMS when Phase 5 commerce traffic patterns are known.
