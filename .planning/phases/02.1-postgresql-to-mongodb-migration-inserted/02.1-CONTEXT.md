# Phase 2.1: PostgreSQL to MongoDB Migration - Context

**Gathered:** 2026-05-31
**Status:** Ready for planning

<domain>
## Phase Boundary

Pure tech-stack swap: replace every Drizzle/PostgreSQL artifact with Mongoose/MongoDB across the entire backend. All 7 collection schemas, the DB connection plugin, all service query implementations, and all Vitest tests are migrated. FeatureFlags (Phase 1) is also in scope — success criterion 3 requires drizzle-orm and pg to be fully removed from all package.json files, which means no Drizzle code may remain anywhere.

No new features, no new API endpoints, no changes to API response shapes. When the migration is done, every Category Engine endpoint returns byte-for-byte identical responses against MongoDB that it previously returned against PostgreSQL.

</domain>

<decisions>
## Implementation Decisions

### ID Strategy
- **D-01:** Primary keys remain **UUID strings stored as `_id`** in all MongoDB collections. MongoDB ObjectId is not used. `packages/contracts` ID types are unchanged. Phase 3+ cross-references (e.g., `category_id` on products) stay UUID strings.
- **D-02:** UUID generation happens in the **service layer** using `crypto.randomUUID()` before each insert. No Mongoose pre-save hooks for ID generation. This is explicit, testable, and matches the intent of Drizzle's `uuid().defaultRandom()`.
- **D-03:** Cross-collection reference fields (e.g., `attribute_definitions.category_id`) are typed as `{ type: String }` in the Mongoose schema — **no `Schema.Types.ObjectId` ref**. No `.populate()` is used; services join by querying directly (same as the current Drizzle pattern).

### Model Structure & Access
- **D-04:** Mongoose model files live in a **centralized `apps/api/src/db/models/`** directory — one file per collection (e.g., `Category.model.ts`, `AttributeDefinition.model.ts`), with a barrel `index.ts`. This mirrors the current `db/schema/` layout and preserves the project's established centralized-schema convention.
- **D-05:** Services **import Mongoose models directly** (e.g., `import { CategoryModel } from '../../db/models/index.js'`). Models are module-level singletons after `mongoose.connect()`. The `db` value is removed from Awilix DI registration — services no longer receive a database instance via constructor injection.
- **D-06:** Replace `apps/api/src/plugins/drizzle.ts` with `apps/api/src/plugins/mongoose.ts`. The plugin calls `mongoose.connect(env.MONGODB_URI)` at startup and registers an `onClose` hook for `mongoose.disconnect()`. **No `fastify.db` decoration** — since services import models directly, the Fastify instance doesn't need to carry a DB reference. Remove `db: asValue(fastify.db)` from `container.ts`.

### Test Infrastructure
- **D-07:** All service tests use **`mongodb-memory-server`** — a real in-memory MongoDB instance, no mocking. The complex Drizzle chain mock builders (`makeDbMock`, `makeInsertDbMock`, `makeUpdateDbMock`) are deleted. Tests call real Mongoose methods against real data.
- **D-08:** `mongodb-memory-server` is wired via a **global setup file** referenced from `vitest.config.ts` (`globalSetup`). One `MongoMemoryServer` instance starts before all suites and tears down after all suites. Each test file clears relevant collections in `beforeEach` to isolate state without paying per-file startup cost.

### Field Naming Conventions
- **D-09:** Mongoose schemas use **snake_case field names** matching the current Drizzle column names: `created_at`, `updated_at`, `parent_id`, `sort_order`, `category_id`, etc. This preserves API response shapes byte-for-byte — the web-admin panel (Phase 2) requires zero changes. Use `{ timestamps: false }` on all schemas and define `created_at`/`updated_at` fields manually as `{ type: Date, default: Date.now }`.

### Claude's Discretion
- **Mongoose plugin connectivity test** — Whether to write a `mongoose.test.ts` replacement for `drizzle.test.ts` (plugin registers, connects to memory-server, onClose hook fires) or rely on service tests for implicit coverage. Planner decides.
- **Schema strictness for JSONB fields** — Mongoose schema `strict` setting and type choice for previously-jsonb columns (attribute options list `[{ value, label }]`, merchandising blocks `blocks[]`). Options: `Schema.Types.Mixed` with TypeScript types, or inline sub-schemas. Zod in `packages/contracts` remains the authoritative API-boundary validator regardless of choice.
- **FK integrity enforcement location** — PostgreSQL's `ON DELETE RESTRICT` on `categories.parent_id` is gone. Whether the "no orphan subtrees" check lives in the service layer (consistent with depth guard in CategoryService) or in a Mongoose pre-remove hook. Planner picks the most testable approach.

</decisions>

- **Awilix DI registration**: Services registered as `asClass().singleton()` in `container.ts`. After migration, services no longer receive `db` but all other constructor injections (redis, logger, env) remain.

- **Zod validation**: Request/response validation stays in `packages/contracts` Zod schemas. Mongoose schema validation is secondary; Zod at the API boundary is the authoritative validator.
- **Redis cache pattern**: CategoryService's Redis tree cache (from Phase 2) is unchanged. Only the DB query underneath the cache invalidation logic changes.

<specifics>
## Specific Ideas

- The existing `requiresSsl()` helper in `drizzle.ts` is PostgreSQL-specific — delete it with the plugin. MongoDB TLS is handled via the connection string (`mongodb+srv://` or `?tls=true`).
- The `CategoryService` depth guard (service-layer enforcement, D-01 from Phase 2) is the right model for all integrity checks that PostgreSQL previously enforced via FK constraints. The planner should apply this pattern to orphan-subtree prevention as well.
- `crypto.randomUUID()` is available in Node.js 22 LTS without any import — no uuid package needed.
- The `db:generate` and `db:migrate` scripts in `apps/api/package.json` are drizzle-kit commands — remove them. No equivalent needed; MongoDB is schemaless and Mongoose creates indexes on first connect.
- Mongoose `_id` must be set to `String` type explicitly in schemas (not `ObjectId`) to store UUID strings: `_id: { type: String, default: () => randomUUID() }` — but per D-02, the service provides the value rather than relying on the default.

</specifics>

<deferred>
## Deferred Ideas

- **drizzle.config.ts** (`DATABASE_DIRECT_URL` pattern from Phase 10) — Phase 10 added this file for Neon's direct connection support. It's deleted in Phase 2.1 along with drizzle-kit. If the project ever needs direct connection configuration for MongoDB, it belongs in the env config, not a separate config file.
- **MongoDB replica set for transactions** — MongoDB multi-document ACID transactions require a replica set. Phase 5 (Commerce Core) needs atomic inventory reservation + order placement. This is a Phase 5 concern; Phase 2.1 uses single-document operations only (Category Engine has no multi-document transaction requirements).
- **Connection pooling config** — Mongoose's default connection pool (5 connections) is fine for Phase 2.1. Tuning `maxPoolSize`, `minPoolSize`, and `serverSelectionTimeoutMS` is a Phase 5+ concern when commerce traffic patterns are known.

None of the above add scope to Phase 2.1.

</deferred>
